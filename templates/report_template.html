Overview

📋 Risk Evaluation Methodology

This audit uses industry-standard security best practices to evaluate AWS Security Groups. Each finding is classified based on the potential impact and exploitability of the security misconfiguration.

🔴 CRITICAL Severity

Definition: Security configurations that pose immediate, severe risk of data breach or system compromise.

Criteria:

  • Database Ports Exposed to Internet (0.0.0.0/0): Ports 3306 (MySQL), 5432 (PostgreSQL), 1433 (SQL Server), 27017 (MongoDB), 6379 (Redis), 9200 (Elasticsearch)
  • All Protocols/Ports Open: Security group allows ALL traffic from the internet
  • Telnet (Port 23): Insecure, unencrypted protocol exposed to internet

Why Critical: These configurations allow direct access to sensitive data stores or complete system access from the internet. Attackers actively scan for and exploit these exposures. Data breaches, ransomware, and complete system compromise are likely outcomes.

Required Action: Immediate remediation required - typically within 24 hours.

🟠 HIGH Severity

Definition: Security configurations that allow unauthorized administrative access or expose sensitive services.

Criteria:

  • SSH (Port 22) Exposed to Internet: Remote shell access available from anywhere
  • RDP (Port 3389) Exposed to Internet: Windows Remote Desktop accessible from internet
  • Management Ports: VNC (5900), WinRM (5985/5986) exposed to 0.0.0.0/0
  • Other Risky Service Ports: FTP (20/21), SMTP (25), SMB (445), HTTP (80), HTTPS (443) without proper controls

Why High: These ports provide administrative or service-level access. While not immediately critical like databases, they are commonly targeted in attacks. Brute force attacks, credential stuffing, and exploitation of known vulnerabilities are common attack vectors.

Required Action: Remediation recommended within 7 days. Implement VPN, bastion hosts, or AWS Systems Manager Session Manager.

🟡 MEDIUM Severity

Definition: Security configurations that increase attack surface but require additional conditions for exploitation.

Criteria:

  • Web Services Without WAF: HTTP/HTTPS exposed without CloudFront or ALB protection
  • Non-Critical Service Ports: Other application ports exposed to internet
  • Overly Broad CIDR Ranges: /8 or /16 ranges when more specific ranges are appropriate

Why Medium: While these configurations expose services, the risk is mitigated by application-level security controls or the nature of the service. However, they still increase attack surface and may facilitate reconnaissance or denial of service attacks.

Required Action: Review and remediate within 30 days. Implement defense-in-depth controls.

🔵 LOW Severity

Definition: Security configurations that represent best practice violations but pose minimal immediate risk.

Criteria:

  • Permissive Egress Rules: All outbound traffic allowed (common and often necessary)
  • Documentation Issues: Missing descriptions on rules

Why Low: These are generally acceptable configurations but may complicate incident response or data exfiltration detection.

Required Action: Review as part of regular security maintenance.

ℹ️ INFORMATIONAL

Definition: Opportunities for optimization and improved security posture.

Criteria:

  • Unused Security Groups: No attached resources (EC2, RDS, ELB, etc.)

Why Informational: Unused security groups don't pose security risks but create management overhead and potential confusion. Removing them reduces complexity and the chance of misconfiguration.

Required Action: Clean up as part of regular maintenance to reduce clutter.

📊 Risk Scoring Methodology:

Each security group rule is automatically evaluated against this framework. Severity is determined by:

  1. Port Classification: Is it a critical database port, management port, or standard service port?
  2. Source IP Range: Is it exposed to the entire internet (0.0.0.0/0 or ::/0)?
  3. Protocol Type: Is it ALL protocols or specific protocol?
  4. Resource Attachment: Are there active resources using this security group?

Industry Standards Referenced: CIS AWS Foundations Benchmark, NIST Cybersecurity Framework, AWS Security Best Practices, OWASP Cloud Security Guidelines
